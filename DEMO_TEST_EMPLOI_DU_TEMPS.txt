================================================================================
  RAPPEL DU FLUX DE PLANIFICATION (ce que la démo va prouver)
================================================================================

  Quand je planifie une séance, mon service fait 6 vérifications/actions :
    1. Valider les données envoyées
    2. Vérifier que le COURS existe        → appel HTTP vers catalogue (8081)
    3. Vérifier que la SALLE est libre      → appel HTTP vers locaux (8082)
    4. Détecter un CONFLIT horaire          → vérification en base locale
    5. RÉSERVER la salle (OCCUPE)           → appel HTTP PATCH vers locaux
    6. ENREGISTRER la séance                → INSERT en base emploi_db

================================================================================
  TEST 5 — CONFLIT HORAIRE (créneau qui chevauche → 409) ★★★ MOMENT FORT
================================================================================

  BUT : démontrer la détection de conflit (l'algorithme HeureUtil).

  CONTEXTE : la salle 1 est déjà prise LUNDI de 08:00 à 10:00.
  On essaie de planifier une AUTRE séance dans la MÊME salle, MÊME jour,
  sur un créneau QUI CHEVAUCHE (09:00 à 11:00).

  COMMANDE :
    curl -i -X POST http://localhost:8083/api/emploi-du-temps \
      -H "Content-Type: application/json" \
      -d '{"coursId":2,"localId":1,"jour":"LUNDI","heureDebut":"09:00","heureFin":"11:00","semestre":"S2-2026","type":"TD"}'

  RÉSULTAT ATTENDU : HTTP 409 Conflict :
    {"erreur":"Conflit de planification : la salle est déjà occupée sur ce créneau horaire","statut":409}

  CE QUE JE DIS AU PROF :
  « Ici je démontre le cœur technique : mon algorithme détecte que le créneau
   09:00-11:00 chevauche le créneau 08:00-10:00 sur la même salle le même jour.
   La formule est d1 < f2 ET d2 < f1. Je retourne un 409 Conflict, code HTTP
   spécifique aux conflits de ressource. »

================================================================================
  TEST 6 — PAS DE CONFLIT SI CRÉNEAU CONSÉCUTIF (→ 201)
================================================================================

  BUT : prouver que l'algorithme est intelligent (créneaux qui se suivent OK).

  CONTEXTE : salle 1 prise de 08:00 à 10:00. On planifie de 10:00 à 12:00.
  Les créneaux se SUIVENT mais ne se CHEVAUCHENT PAS → autorisé.

  COMMANDE :
    curl -X POST http://localhost:8083/api/emploi-du-temps \
      -H "Content-Type: application/json" \
      -d '{"coursId":2,"localId":1,"jour":"LUNDI","heureDebut":"10:00","heureFin":"12:00","semestre":"S2-2026","type":"TD"}'

  RÉSULTAT ATTENDU : HTTP 201 Created (pas de conflit, créneaux consécutifs) :
    {"id":2,"coursId":2,"localId":1,"jour":"LUNDI","heureDebut":"10:00",...}

  CE QUE JE DIS AU PROF :
  « Mon algorithme distingue chevauchement et succession : 10:00-12:00 commence
   exactement quand 08:00-10:00 finit, donc pas de conflit. La condition
   d2 < f1 (600 < 600) est fausse, donc aucun chevauchement détecté. »

================================================================================
  RÉCAPITULATIF DES CODES HTTP DE MON SERVICE
================================================================================

    201 Created     → séance planifiée avec succès
    204 No Content  → séance supprimée
    400 Bad Request → données invalides / cours absent / salle indisponible
    404 Not Found   → séance ou jour introuvable
    409 Conflict    → conflit horaire (MA signature) ★
    500 Internal    → erreur technique (réseau, base)
